## ADDED Requirements

### Requirement: Try/catch во всех async функциях store
Каждая асинхронная функция в Pinia store, выполняющая API вызов, ДОЛЖНА быть обёрнута в try/catch блок. При ошибке функция ДОЛЖНА логировать ошибку в консоль (только в dev режиме) и возвращать `null` или пустой массив в зависимости от контекста.

#### Scenario: Успешный API вызов
- **WHEN** API вызов завершается успешно
- **THEN** функция возвращает данные из `response.data`

#### Scenario: Ошибка API вызова
- **WHEN** API вызов выбрасывает ошибку (сеть, 4xx, 5xx)
- **THEN** ошибка логируется в dev режиме и функция возвращает `null`

### Requirement: Пользовательские уведомления об ошибках
При ошибке обновления данных (PUT/POST запросы) система ДОЛЖНА выводить уведомление через Naive UI notification компонент с описанием ошибки. Уведомление ДОЛЖНО быть видимым минимум 3 секунды.

#### Scenario: Ошибка при обновлении сотрудника
- **WHEN** `updateUser()` завершается ошибкой
- **THEN** выводится error notification с текстом ошибки

#### Scenario: Ошибка при загрузке фото
- **WHEN** `uploadFile()` завершается ошибкой
- **THEN** выводится error notification с текстом ошибки

### Requirement: Loading состояния во время API вызовов
Store ДОЛЖЕН поддерживать loading состояния (`ref(false)`) для длительных операций. Компоненты ДОЛЖНЫ иметь возможность отображать disabled состояние кнопок и спиннеры на основе этих состояний.

#### Scenario: Начало загрузки
- **WHEN** вызывается `getUsers()`
- **THEN** `isLoading` устанавливается в `true`

#### Scenario: Завершение загрузки
- **WHEN** `getUsers()` завершается (успех или ошибка)
- **THEN** `isLoading` устанавливается в `false`
